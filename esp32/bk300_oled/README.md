# BK300 OLED monitor (ESP32-C3 / ESP32 + SSD1306)

То же, что `esp32/bk300_oneshot_sleep.ino`, но c выводом **текущего напряжения**
и **состояния BLE-связи** на 128×64 OLED-дисплей SSD1306 (I²C, `0x3C`).

## Внешний вид

```
┌───────────────────────────────┐
│ BK300: Live                   │   <-- статус (size 1)
│-------------------------------│
│                               │
│   12.45 V                     │   <-- крупное напряжение (size 3 + 2)
│                               │
│  47s             rx 1s        │   <-- uptime + «возраст» последнего значения
└───────────────────────────────┘
```

Статусы, которые могут быть в верхней строке:

| Статус | Когда показывается |
| --- | --- |
| `Init BLE` | в `setup()`, до первой попытки соединения |
| `Scanning...` | активный скан BLE |
| `Connecting` | scan нашёл устройство, идёт `connect(addr)` |
| `Reconnecting` | пробуем cached MAC из `Preferences` |
| `Connected` | TCP-аналог: GATT-сессия открыта |
| `Kicking...` | шлём `0B06 → 0100 → 0B01 → 0B08 → 0B0B` |
| `Live` | kick прошёл, идёт периодический `0B0B` и приходят `4B0B` |
| `Disconnected` | потеряли GATT-сессию; начался backoff |
| `Wait Ns` | в backoff между переподключениями (N — секунд осталось) |
| `Not found` | скан не нашёл BK300 в эфире |
| `No FFF0 svc` / `No FFF1/FFF2` / `No CCCD` | устройство нашлось, но GATT не такой |

## Зависимости (Arduino IDE → Library Manager)

- **Adafruit GFX Library**
- **Adafruit SSD1306**
- **штатная** `BLEDevice.h` из `arduino-esp32` (НЕ ставить отдельный
  NimBLE-Arduino — будет конфликт заголовков).

## Пины I²C по умолчанию

```
#define I2C_SDA  8     // ESP32-C3 dev-board: GPIO 8
#define I2C_SCL  9     // ESP32-C3 dev-board: GPIO 9
```

Для ESP32 (классический WROOM/WROVER) поменяй на `21` / `22`. Для ESP32-S3
обычно `8` / `9` тоже подходит, но смотри по плате. Адрес SSD1306 у
большинства китайских модулей — `0x3C`, реже `0x3D`.

Если экран смонтирован «вверх ногами», правится в одной строке:

```c
#define OLED_ROTATION 2   // 0/1/2/3 — повороты на 90°
```

## Settings в Arduino IDE

- Tools → **Partition Scheme: Huge APP (3MB No OTA / 1MB SPIFFS)** —
  Bluedroid + GFX могут не влезть в дефолтный 1.4 MB партишен.
- Tools → CPU Frequency: 160 / 240 MHz; Flash Size: 4 MB; Flash Mode: QIO;
  Flash Frequency: 80 MHz.
- Tools → Board: ESP32C3 Dev Module (или твоя плата).

## Сборка

1. Открыть `bk300_oled/bk300_oled.ino` в Arduino IDE.
2. Поставить нужную плату и партишен (см. выше).
3. Поправить `I2C_SDA` / `I2C_SCL` / `OLED_ROTATION` под свою железку,
   если отличаются.
4. Прошить.

## Что должно быть на экране при успешной работе

После прошивки за ~5–10 секунд статус сменится:

```
Init BLE → Scanning... → Connecting → Connected → Kicking... → Live
```

И после `Live` крупная строка перестанет быть `--.-- V`, появится реальное
напряжение, например `12.45 V`. В правом нижнем углу — счётчик секунд с
последнего успешно полученного `4B0B`. Если он растёт выше 5–10 секунд,
значит связь «зависла» — обычно происходит ребоут или backoff на следующем
тике.

## Если не работает

1. **Экран чёрный, серийный лог пишет `SSD1306 allocation failed`**:
   проверь пины `I2C_SDA` / `I2C_SCL`, адрес `SCREEN_ADDRESS` (`0x3C` /
   `0x3D`), напряжение питания на модуле (3.3 V).
2. **Экран есть, статус застрял на `Scanning...`**: BK300 рядом, но не
   рекламирует FFF0 / имя BK300. Проверь, что прибор на питании и ничего
   другое к нему не подключено по BLE одновременно (он умеет только одну
   сессию).
3. **`Live` приходит, но `--.-- V` не меняется (rx s растёт)**: см.
   `docs/spec.md`, раздел «КРИТИЧНО: `0B01` ожидает реальный Unix-epoch».
   В этом скетче уже стоит fallback-epoch `1772582400`, но если он со
   временем устарел и устройство снова станет «капризничать» — поднять
   `BK300_FALLBACK_EPOCH` в исходнике, либо поднять SNTP/NTP, чтобы
   `time(NULL)` отдавал настоящее время.
