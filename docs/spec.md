# BK300 BLE protocol specification

Документ описывает BLE-протокол монитора BK300 на основе данных nRF Connect и декомпиляции приложения BKmonitor (`com.jiawei.batteryonline`).

Статус: рабочая спецификация. GATT-профиль, формат команд и CRC подтверждены кодом приложения. Режим волны (`WaveformAcitivty`): разбор `connectStr`, ответ `4501` и второй (компактный) поток — по коду. Семантика части полей и точная пара «команда → первый кадр ответа» (например, только `0100`) — по логам с прибора.

## Устройство

- Имя устройства: `BK300`
- MAC из nRF Connect: `B3:00:69:77:51:08`
- BLE-режим для ESP32: `central` / `client`
- Протокол обмена: BLE GATT, не SPP/RFCOMM

## GATT-профиль

Основной проприетарный сервис:

| Назначение | UUID |
| --- | --- |
| Primary service | `0000fff0-0000-1000-8000-00805f9b34fb` (`0xFFF0`) |
| Notify characteristic | `0000fff1-0000-1000-8000-00805f9b34fb` (`0xFFF1`) |
| Write characteristic | `0000fff2-0000-1000-8000-00805f9b34fb` (`0xFFF2`) |

Свойства характеристик:

- `0xFFF1`: `NOTIFY`. На неё нужно подписаться через CCCD (`0x2902`) значением `01 00`.
- `0xFFF2`: `WRITE NO RESPONSE`. В неё отправляются команды.

Приложение BKmonitor не жёстко привязано к `FFF1/FFF2`: оно ищет сервис, в UUID которого есть `fff0`, затем выбирает notify-характеристику по флагу `PROPERTY_NOTIFY` (`0x10`), а write-характеристику по `PROPERTY_WRITE_NO_RESPONSE` (`0x04`). На ESP32 лучше делать так же, но для BK300 из nRF Connect ожидаемые UUID именно `FFF0/FFF1/FFF2`.

## Общий порядок работы

1. Включить BLE на ESP32 в режиме central.
2. Найти устройство по MAC `B3:00:69:77:51:08` или имени `BK300`.
3. Подключиться к устройству.
4. Выполнить service discovery.
5. Найти сервис `0xFFF0`.
6. Найти `0xFFF1` и включить notify.
7. Найти `0xFFF2` и писать команды через write without response.
8. Отправить первичную команду опроса, обычно `0100`.
9. Для периодического обновления отправлять `0301` или использовать команды, описанные ниже, в зависимости от режима.

## Формат кадра

Все команды и ответы передаются бинарными байтами. В приложении они часто представлены как hex-строка.

Обычный кадр (формат полей):

```text
SS SS LL LL CC CC [payload...] CRC_LO CRC_HI 0D 0A
```

Поля:

| Поле | Размер | Описание |
| --- | ---: | --- |
| `SS SS` | 2 байта | начало кадра |
| `LL LL` | 2 байта | длина little-endian |
| `CC CC` | 2 байта | код команды или ответа |
| `payload` | N байт | данные команды или ответа |
| `CRC_LO CRC_HI` | 2 байта | CRC little-endian |
| `0D 0A` | 2 байта | конец кадра |

### Важное уточнение по префиксу кадра (живые логи)

В исходной декомпиляции отправляемые команды собираются с префиксом `40 40`. Однако по живому логу BK300 (notify с `FFF1`) подтверждено, что **входящие** кадры могут начинаться с **`24 24`** (ASCII `$$`), при этом структура `LL LL`, `CC CC`, CRC и окончание `0D 0A` сохраняются.

Пример полного входящего кадра (notify):

```text
24 24 36 00 43 01 42 4b 33 30 30 00 ... 33 e6 0d 0a
^^ ^^ ^^ ^^ ^^ ^^
SS SS LL LL CC CC
```

Интерпретация примера:

- `SS SS = 24 24`
- `LL LL = 36 00` → длина 54 байта
- `CC CC = 43 01` → тип/ответ `4301`
- в конце кадра: `CRC_LO CRC_HI` затем `0D 0A` (в примере: `33 e6 0d 0a`)

Длина `LL LL` в отправляемых командах считается так:

```text
length = payload_size_bytes + 10
```

То есть длина включает весь кадр: `40 40` + `LL LL` + команда + payload + CRC + `0D 0A`.

Пример команды `0100` без payload:

```text
40 40 0A 00 01 00 <crc_lo> <crc_hi> 0D 0A
```

Особый случай: команда `0301` в приложении зашита как фиксированный кадр:

```text
40 40 0A 00 03 01 33 D3 0D 0A
```

## CRC

CRC считается по байтам от **начала кадра** (`SS SS`) до конца payload включительно. Поля CRC и `0D 0A` в расчёт не входят.

Алгоритм из приложения:

- начальное значение: `0xFFFF`
- табличный FCS/CRC, таблица `FCSTAB`
- после обработки всех байтов результат инвертируется через XOR с `0xFFFF`
- в кадр пишется little-endian: сначала младший байт, потом старший

Псевдокод:

```c
uint16_t bk300_crc(const uint8_t *data, size_t len) {
    uint16_t fcs = 0xFFFF;

    for (size_t i = 0; i < len; i++) {
        fcs = (fcs >> 8) ^ FCSTAB[(data[i] ^ fcs) & 0xFF];
    }

    return fcs ^ 0xFFFF;
}
```

Для проверки реализации можно свериться с фиксированной командой `0301`:

```text
data_for_crc = 40 40 0A 00 03 01
crc          = 33 D3
full_frame   = 40 40 0A 00 03 01 33 D3 0D 0A
```

## Сборка команды

Функция сборки кадра:

```text
build_command(command, payload):
    length = payload.length + 10
    body = 40 40 + uint16_le(length) + uint16_or_raw_command(command) + payload
    crc = crc16(body)
    return body + uint16_le(crc) + 0D 0A
```

В коде приложения команда передаётся как 4 hex-символа, например `0100`, `0B01`, `0301`. Эти два байта отправляются в кадр в том же порядке, как записаны в hex-строке.

Для команд с payload payload также передаётся как hex-строка. Например смена BLE-имени:

```text
command = 0C02
payload = <len_utf8_name_1_byte> + <utf8_name_as_hex>
```

## Приём уведомлений

Уведомления приходят кусками по `0xFFF1`. Один логический кадр может быть разбит на несколько notify-пакетов, поэтому нужно накапливать байты до окончания `0D 0A`.

Алгоритм приёма:

1. Каждый notify-пакет добавить в буфер.
2. Если буфер короче 10 байт, ждать дальше.
3. Когда буфер заканчивается на `0D 0A`, разобрать кадр.
4. Проверить префикс `SS SS`:
   - `40 40` (как в отправляемых командах из приложения)
   - **или** `24 24` (подтверждено живыми notify от BK300)
5. Проверить длину.
6. Проверить CRC.
7. Код ответа читать из байтов `4..5`.
8. Payload начинается с байта `6`.

В приложении код ответа берётся как hex-подстрока `connectStr.substring(8, 12)`, то есть из байтов `4..5`.

### Замечание: как приложение детектирует «полный» кадр в `connectStr`

В `WaveformAcitivty` и `MainActivityNew` (и др.) `connectStr` — это **склейка** подстрок `HexUtil.formatHexString(data, false)` из **каждого** `onCharacteristicChanged`, без разделителей.

Признак, что кадр собран (ветка волны / главного экрана), в коде такой:

- взять `connectStr` без **последних 4** hex-символов: `connectStr.substring(0, connectStr.length() - 4)` (последние 2 байта кадра в бинарнике, в виде 4 hex-цифр, трактуются как **CRC**);
- у этой усечённой hex-**строки** суффикс (буквально символы) `0d0a` (то есть в конце **бинарного** кадра идут байты `0D 0A` *до* отбрасываемого хвоста — так реализована проверка готовности).

Итог для ESP32: при накоплении notify **не** обязательно дублировать вариант со `substring(...-4)` из Android-кода, достаточно дождаться полного кадра по **байтам** `... CRC_LO CRC_HI 0D 0A` и затем проверить CRC.

Наблюдение по живому логу BK300: кадр заканчивается **`CRC_LO CRC_HI 0D 0A`** (т.е. как в `HexUtil.sendWriteData` при отправке), пример: `... 33 e6 0d 0a`.

## Режим «волна» (`WaveformAcitivty`)

После `0100` в этом экране **ещё** используются обычные кадры `40 40 …`. Обработчик `onCharacteristicChanged` ведёт себя **по-разному** в зависимости от флага `isReceive` (и собранной строки).

### 1) Пока `isReceive == false`: только полные кадры `40 40 …`

- Накапливается `charList` → `ArrayToStringUtils.getCharlistString` → `connectStr`.
- Код «типа пакета» (два байта ответа после длины): hex-подстрока **`connectStr.substring(8, 12)`** (4 hex-символа, байты 4..5 в кадре).
- Следом один байт статуса для части команд: **`connectStr.substring(12, 14)`** (1 байт = 2 hex-символа).

Важно: в **этом** обработчике **нет** отдельной ветки `if (код == "4100")` для ответа на `0100`. Сценарий «включить поток сырых выборок» привязан к ответу **`4501`**, а не к явному `4100`.

| Подстрока `8..12` (hex) | Смысл по коду (Waveform) |
| --- | --- |
| `4501` | «Разрешение» режима потока: `12..14` = `00` → `isReceive = true` и `handler` сообщение `2`; `02` → сообщение `7`; иначе `isReceive = false`, сообщение `3`. |
| `4203` | Флаг OTA/обновления, как на главном экране. |
| `4502` | Сброс/очистка списка значений (лог + `valueList.clear`). |
| `4204` | Разбор длины/порога, проверка CRC в конце кадра, ветвление `handler` 66 / 89. |
| `4301` | Снятие флага запроса, поля: `12..32` — имя, `32..52` — версия IAP, `52..72` — строка в `message.obj` (MCU/версия), далее `handler` 81. |

### 2) Когда `isReceive == true`: короткие notify **без** обёртки `40 40 …`

Пока `isReceive` истинно, `HexUtil.formatHexString` от **одного** notify **не** обязан содержать `40 40` / `0d0a` — смотрят **длину** hex-строки (2, 4, 6, 8 символов или кратно 4) и тянут `getDaduan(подстрока)/100.0` как **напряжение, В** (см. `BigDecimal` + деление на 100).

- Для пачек, кратных 4 hex-символам, путь «разбить на группы по 2 байта» выполняется **только если** в строке **нет** подстрок `2424` и `0d0a` (т.е. это не кусок «фичерного» кадра в том же notify).

Это **второй протокол поверх той же характеристики `FFF1`**: сначала — полные `40 40` кадры, после `4501`+`00` — поток **компактных** сэмплов.

#### 2.1) Подтверждённый формат сэмплов напряжения

Ключевой факт из кода `WaveformAcitivty`: **каждый сэмпл — это 2 байта (U16 little-endian), масштабирование \(V = \text{U16} / 100.0\)** с округлением до 2 знаков.

То есть на проводе приходит целое число в “сотых долях вольта”:

```text
raw_u16_le = <lo><hi>
voltage_v  = raw_u16_le / 100.0
```

Примеры:

```text
LE: B8 0B -> 0x0BB8 -> 3000 -> 30.00 V
LE: C4 09 -> 0x09C4 -> 2500 -> 25.00 V
```

#### 2.2) Как приложение обрабатывает фрагментацию/агрегацию notify в потоке

В режиме потока одно уведомление `FFF1` может принести 1+ сэмплов, либо “разорванный” сэмпл. Логика `WaveformAcitivty` (в терминах **hex-строки** одного notify) такая:

- **длина 2 hex-символа (1 байт)**: байт буферизуется в `tempList`, следующий notify дополняет его до 2 байт.
- **длина 4 hex-символа (2 байта)**: один сэмпл.
- **длина 6 hex-символов (3 байта)**:
  - если уже есть “хвост” 1 байт в `tempList`, то собирается 1‑й сэмпл из (хвост + первый байт текущего notify), а оставшиеся 2 байта — 2‑й сэмпл;
  - иначе пакет отбрасывается/очищается буфер (защита от рассинхронизации).
- **длина 8 hex-символов (4 байта)**: два сэмпла.
- **длина кратна 4 hex-символам (т.е. кратна 2 байтам)**: режется на куски по 4 hex-символа (2 байта) → каждый кусок = один сэмпл.

В ветке “кратно 4” есть защита: пакет **не** парсится как массив сэмплов, если в нём встречаются `2424` или `0d0a` (чтобы не перепутать с кусками обычных кадров `40 40 ...`).

Вывод для реализации на ESP32/PC:

- Принимать поток как последовательность байтов и собирать **U16 little-endian**.
- Держать буфер на 1 байт (если notify оборвал сэмпл).
- Быть готовым, что один notify содержит сразу несколько сэмплов.

#### 2.3) Ограничение по максимуму (`maxVoltage`) и кадр `4204`

`WaveformAcitivty` отбрасывает сэмплы, у которых `raw_u16_le > maxVoltage` (в тех же “сотых долях вольта”).

`maxVoltage` задаётся в обработке полного кадра типа **`4204`**: приложение читает поле `connectStr.substring(14, 18)` (2 байта) и пишет в `maxVoltage`. Если значение равно `0`, приложение принудительно ставит большой предел (фактически выключая фильтр).

Подтверждение по живому логу BK300 (входящий кадр с префиксом `24 24`):

```text
24 24 1a 00 42 04 00 08 07 08 07 00 00 20 80 1e 00 20 98 1b 00 20 58 18 0d 0a
SS SS LL LL CC CC ST MV_LO MV_HI ...
```

Разбор:

- `SS SS`: `24 24`
- `LL LL`: `1a 00` → длина кадра 26 байт
- `CC CC`: `42 04` → тип `4204`
- `ST` (status): `00` (в `WaveformAcitivty` это `connectStr.substring(12, 14)`)
- `maxVoltage`: `08 07` little-endian → `0x0708 = 1800` → **18.00 V**
- CRC находится в последних двух байтах **перед** `0d0a`: в примере `58 18` (little-endian CRC = `0x1858`)

Примечание: остальные поля `4204` пока не подписаны (требуются дополнительные логи/сопоставление с поведением прибора), но самого `maxVoltage` достаточно, чтобы корректно фильтровать поток, как это делает приложение.

### 3) Команда `0100` и ожидаемый «ответ» в этом месте

- В `WaveformAcitivty` команда `0100` **отсылается** (инициализация).
- Прямой пары `0100` → `4100` **в обработчике нет**; чтобы выйти на поток, приложение **использует** ответы кадра с кодом **`4501`**, а **не** проверяет `4100` в этом файле.
- Следовательно, **по коду** формат, который **явно** открывает разбор вольт-выборок — это **полный** кадр, заканчивающийся согласно проверке `connectStr`, с типом `4501` и первым байтом payload (hex `12..14`) равным `00` (успех).

Для натурного логирования на ESP32: после `0100` записать **все** дошедшие кадры; отдельно зафиксировать, **приходит ли** `4100` или сразу `4501` / `4301` / `4B…` — это снимет неоднозначность, которую в декомпиляте в одном `if` не закрыть.

Наблюдение по живому логу: на прошивке, с которой сняты логи, после последовательности `0301 → 0204 → 0100 → 0301` приходят `4301` и `4204`, но **кадр `4501` пока не наблюдался**. Это означает, что:

- либо поток Waveform на этом экземпляре/прошивке включается другой командой/последовательностью,
- либо `4501` приходит реже и его нужно ловить при более длительном/частом опросе,
- либо прибор не поддерживает поток в текущем режиме.

## Команды и ответы

В ответах старший байт часто отличается от команды: например команда `0B01` ожидает ответ `4B01`, команда `0C02` ожидает `4C02`. Это похоже на правило: первый байт ответа = первый байт команды + `0x40`.

Подтверждённые по коду команды:

| Команда | Ожидаемый ответ | Назначение по коду приложения |
| --- | --- | --- |
| `0100` | в `WaveformAcitivty` нет сопоставления с `4100`; ответ `4501` открывает поток | инициализация, часто в цепочке с другими командами; однозначный «ответ на только 0100» в коде не выделен |
| (ответ) | `4501` + байт в hex `12..14` | Waveform: `00` / `02` / иначе — ветвление, при `00` — поток U16/100 вольт в коротких notify |
| `0301` | встречается в тестовых режимах | короткий периодический запрос, фиксированный кадр |
| `0201` | `4201` вероятно | запрос со страницы About |
| `0203` | `4203` | проверка/статус обновления; payload `00` означает нормальное состояние |
| `0204` | `4204` вероятно | используется в стартовом/волновом тесте |
| `0302` | `4302` вероятно | настройка/запись параметров |
| `0303` | `4303` вероятно | настройка/запись параметров |
| `0304` | `4304` вероятно | OTA/file-передача |
| `0305` | `4305` вероятно | настройка/запись параметров |
| `0601` | `4601` вероятно | команда из stand test с числовым payload |
| `0603` | `4603` вероятно | команда из stand test |
| `0703` | `4703` вероятно | команда из start test |
| `0901` | `4901` вероятно | production/test режим |
| `0A03` | `4A03` вероятно | удаление/прогресс/операция с ID |
| `0B01` | `4B01` | запрос, после ответа приложение запрашивает следующие данные |
| `0B02` | `4B02` вероятно | команда из диагностики/сбора данных |
| `0B03` | `4B03` вероятно | команда из диагностики/сбора данных |
| `0B04` | `4B04` вероятно | команда из диагностики/сбора данных |
| `0B05` | `4B05` вероятно | команда из диагностики/сбора данных |
| `0B06` | `4B06` вероятно | команда с payload параметров батареи |
| `0B07` | `4B07` вероятно | настройка значения в settings |
| `0B08` | `4B08` | чтение состояния флага; payload `00` = выключено, иначе включено |
| `0B09` | `4B09` вероятно | команда из диагностики/сбора данных |
| `0B0A` | `4B0A` вероятно | команда из диагностики/сбора данных |
| `0B0B` | `4B0B` | текущий статус главного экрана / уведомления |
| `0C02` | `4C02` | смена BLE-имени; payload ответа `00` = успех |

### Ответ `4B0B`

Приложение использует `4B0B` как статусный ответ главного экрана:

| Смещение в payload | Размер | Использование в приложении |
| --- | ---: | --- |
| `0..1` | 2 байта | **напряжение**: U16 little-endian в сотых долях вольта, \(V = U16 / 100.0\) |
| `2` | 1 байт | статус уведомления; `0` включает notification в Android |
| `3` | 1 байт | статус иконки батареи; `0` скрывает иконку, другие значения выбирают картинку |

Подтверждение по живому логу BK300:

```text
24 24 0e 00 4b 0b cc 04 01 01 45 28 0d 0a
SS SS LL LL CC CC  [payload......] CRC  0d0a
```

Разбор:

- `LL LL`: `0e 00` → длина кадра 14 байт
- `CC CC`: `4b 0b` → ответ `4B0B`
- payload:
  - `cc 04` little-endian → `0x04CC = 1228` → **12.28 V**
  - `01` → флаг уведомления (в приложении используется как включить/выключить уведомление)
  - `01` → статус иконки батареи (в приложении выбирает картинку)
- CRC: `45 28` (перед `0d0a`)

Итого: если цель устройства — “только напряжение”, то `0B0B → 4B0B` — самый простой и устойчивый путь получения текущего напряжения без Waveform-потока.

### Ответ `4301`

Приложение разбирает ответ `4301` так:

| Payload bytes | Назначение |
| --- | --- |
| `0..9` | имя устройства в hex/ASCII/UTF-8 |
| `10..19` | версия IAP/firmware |
| `20..29` | дополнительное поле, отправляется дальше в обработчик |

## Минимальный сценарий получения данных

Практический минимальный сценарий для ESP32:

1. Подключиться к BK300.
2. Подписаться на notify `0xFFF1`.
3. Отправить `0100` без payload в `0xFFF2`.
4. Накопить ответы до `0D 0A`, проверить CRC и залогировать полные кадры.
5. Отправить `0301` фиксированным кадром `40400A00030133D30D0A`.
6. Отправить `0B0B` (опрос текущего статуса/напряжения). В ответе `4B0B` первые 2 байта payload — **напряжение** в формате U16LE/100 (см. раздел «Ответ `4B0B`).
7. Для полной расшифровки физики полей записать несколько кадров при известных состояниях: без нагрузки, зарядка, запуск двигателя/нагрузка, разные напряжения.
8. Для режима волны: после `0100` (и остальной цепочки сценария) ожидаем полный кадр с `4501`; при `12..14` = `00` дальше приходят **короткие** notify с группами по 2 байта (см. раздел «Режим волна»), а не только `40 40 … 0D 0A` кадры.

### Минимальный сценарий: Waveform (поток напряжения)

Цель: получить **поток** сэмплов напряжения \(U16 little-endian / 100.0\) в notify на `FFF1`, как это делает экран `WaveformAcitivty`.

#### 0) Подключение и GATT

1. Просканировать BLE и найти устройство по имени `BK300` (или по MAC, если оно стабильно для вашего экземпляра).
2. Подключиться (Central/Client).
3. Выполнить service discovery.
4. Найти проприетарный сервис `FFF0` и характеристики:
   - `FFF1` (Notify)
   - `FFF2` (Write No Response)
5. Включить notify на `FFF1` (подписка через CCCD `0x2902` значением `01 00`).

#### 1) Протокол “обычных” кадров до старта потока

1. Начать принимать notify и накапливать байты до полного кадра `SS SS ... CRC ... 0D 0A`, где `SS SS` это **`40 40` или `24 24`** (см. раздел «Приём уведомлений»).
2. Отправить команду **`0100`** без payload в `FFF2` (кадр с CRC).
3. Если после `0100` **нет ни одного** входящего кадра, отправить **`0301`**: по живому логу BK300 первые ответы (например `4301`) начинают приходить именно после `0301`.
4. (Рекомендуется как в приложении) Отправить **`0204`** и дождаться `4204`, чтобы получить/обновить `maxVoltage` и параметры режима.
5. Дождаться “служебных” ответов (`4301`, `4203`, `4204`, …) и **зафиксировать** их в лог (полезно для отладки и подтверждения префикса/CRC).

Важно: по коду `WaveformAcitivty` “включатель” потока — это не `4100`, а **ответ типа `4501`**.

#### 2) Условие входа в поток (ключевой маркер)

Ожидаемый полный кадр:

- тип/код ответа: `4501` (`connectStr.substring(8, 12)`)
- байт статуса: `00` (`connectStr.substring(12, 14)`)

При выполнении этого условия приложение делает `isReceive = true` и начинает парсить **короткие notify** как поток сэмплов напряжения.

Если статус `02`, приложение уходит в другую ветку (в UI это отдельное сообщение); для протокола Waveform это означает “поток не стартовал”.

#### 3) Парсинг потока сэмплов напряжения

После `4501/00` notify на `FFF1` могут приходить **без** `40 40 ... 0D 0A`. Их нужно разбирать как последовательность байтов:

- каждый сэмпл = 2 байта (U16 little-endian)
- напряжение в вольтах: \(V = U16 / 100.0\)
- учитывайте, что notify может:
  - оборвать сэмпл (1 байт в конце одного notify, 1 байт в начале следующего),
  - содержать сразу несколько сэмплов.

Практический псевдокод:

```text
state = WAIT_FRAMES
byte_buffer = []
sample_half_byte = none

onNotify(bytes):
  if state == WAIT_FRAMES:
    append bytes to frame_buffer
    while have full (40 40 | 24 24) ... CRC ... 0D 0A frame:
      parse frame
      if frame.type == 0x4501 and frame.payload[0] == 0x00:
        state = STREAM
        clear stream buffers

  else if state == STREAM:
    for b in bytes:
      if have_first_byte:
        u16 = first | (b << 8)    // little-endian
        voltage = u16 / 100.0
        emit(voltage)
        clear have_first_byte
      else:
        first = b
        have_first_byte = true
```

#### 4) Порог `maxVoltage` (опционально, но как в приложении)

Приложение отбрасывает сэмплы, превышающие `maxVoltage` (см. «2.3) Ограничение по максимуму»). Для “сырого” логгера можно этот фильтр выключить, но для повторения поведения приложения — применяйте.

Рекомендуемый порядок логирования для обратной разработки:

```text
TX 0100 -> все RX
TX 0301 -> все RX
TX 0B0B -> все RX
TX 0B01 -> все RX
```

Команда `0B0B` в приложении используется сервисом мониторинга, поэтому это хороший кандидат для регулярного получения состояния.

## Набросок ESP32-логики

```cpp
// 1. Scan/connect to BK300 by MAC or name.
// 2. Service:        0000fff0-0000-1000-8000-00805f9b34fb
// 3. Notify char:    0000fff1-0000-1000-8000-00805f9b34fb
// 4. Write char:     0000fff2-0000-1000-8000-00805f9b34fb
// 5. Enable notifications on FFF1.
// 6. Write command frames to FFF2 with writeNoResponse().

void onNotify(const uint8_t* data, size_t len) {
    appendToRxBuffer(data, len);

    while (hasCompleteFrameEnding0d0a()) {
        Frame frame = popFrame();
        if (!frame.hasPrefix4040()) return;
        if (!frame.lengthMatches()) return;
        if (!frame.crcMatches()) return;

        uint16_t response = frame.commandBytesAsSeenOnWire();
        handleBk300Response(response, frame.payload);
    }
}
```

## Важные замечания

- Писать нужно именно в `WRITE NO RESPONSE`. Если библиотека ESP32 позволяет выбрать тип записи, используйте `writeValue(data, len, false)` или аналог.
- Notify может приходить фрагментами, поэтому нельзя парсить каждый notify как отдельный кадр.
- Длина и CRC little-endian.
- MTU можно оставить стандартным; приложение режет длинные записи по 20 байт. Для обычных команд хватает одного BLE-пакета.
- Для стабильного подключения лучше искать сервис/характеристики по UUID, а не только по имени устройства.
- MAC BLE-устройства может меняться, если устройство использует random/private address. По данным nRF Connect для текущего экземпляра: `B3:00:69:77:51:08`.

## Что ещё нужно снять с устройства

Чтобы довести спецификацию до полной расшифровки значений, нужен лог notify с реального BK300:

- сразу после `0100`;
- после `0301`;
- после `0B0B`;
- при разных известных напряжениях батареи;
- при включении/выключении режимов теста в приложении.

После этого можно точно подписать поля payload: напряжение, внутреннее сопротивление, заряд/статус, ошибки и режимы тестирования.
